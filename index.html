Enterprise file ingestion and media processing platform for a daycare management SaaS

This page documents a complete, production-ready architecture for student, staff, guardian, school, and org-level uploads. It includes naming rules, S3 prefix structure, database design, API contracts, image/video/document handling, access control, lifecycle management, and a phased implementation plan.

2 Dedicated S3 buckets: raw + processed
1 Unified file metadata table as source of truth
5 Primary scopes: org, school, students, staff, guardians
3 Processing tracks: image, video, document

1. Executive summary

The recommended design is to separate original uploads from processed derivatives. Clients upload directly to a raw bucket using presigned URLs. S3 events route work through SQS into Lambda-based processors. Images are transformed in Lambda, videos are submitted to MediaConvert, and documents remain preserved as originals with optional previews and text extraction.

Key rule: never overwrite originals. Keep the original file for audit, reprocessing, compliance, quality recovery, and future derivative generation.

Why this fits your SaaS

The system naturally supports per-student isolation, staff and guardian profile media, school-level shared assets, and org-level uploads such as newsletters or admin documents. It also works when a user belongs to multiple schools or exists at the org level without belonging to a school.

What changes from a basic setup

Instead of using loose paths or school-only scoping, every object key begins with org/{org_code} and includes school scope using a real school code or _shared. Every business object is tracked in the database using immutable code values rather than ids.

2. Core design principles

1.
Always org-first. Every object key includes org/{org_code} before anything else.
2.
Always code-based. Use school_code, student_code, file_code, never ids.
3.
Separate originals and derivatives. Raw uploads and processed assets live in different buckets.
4.
Database is source of truth. The app never depends on S3 list operations for business logic.
5.
Paths are deterministic. All keys are built by shared utility functions, never hand-written ad hoc.
6.
Access is policy-driven. Authorize against org, school, subject, and related business entities, not just the path.
7.
Processing is retry-safe. All workers must be idempotent and resilient to duplicate event delivery.

3. Target architecture

Control plane

API service for upload authorization, metadata creation, validation, file listings, secure reads, and business access checks.

Main records live in a unified files table and optional support tables for attachments, activity relationships, newsletter relationships, and processing jobs.

Data plane

Clients upload directly to S3 raw bucket. S3 emits create events. SQS buffers work. Lambda routes processing. MediaConvert handles video transcoding. Processed assets are delivered through CloudFront.

Recommended AWS building blocks

S3 Raw Bucket S3 Processed Bucket SQS Queue Lambda Router Image Processor Lambda Document Processor Lambda MediaConvert CloudFront CloudWatch DLQ

4. End-to-end flow

1Presign request
Client sends org, school, subject, category, filename, size, and content type to backend.
2Create file record
Backend validates scope and writes a files row with status uploading.
3Direct upload
Client uploads to raw bucket using presigned URL for one exact key.
4S3 event
Object-created event lands in SQS to decouple upload spikes from processing.
5Async processing
Lambda inspects file type and routes to image, video, or document flow.
6Ready for delivery
Processed manifest is stored in DB and the app serves allowed variants via CloudFront.
Important: the processed bucket must not trigger the same processing pipeline. Only the raw bucket should create processing events.

5. Scope model

Your system must support both school-linked and org-only users. That means storage and metadata cannot assume that every file belongs to a school. The system therefore always starts at org scope, then optionally adds school scope and subject scope.

Supported scope combinations

Use case org_code school_code subject_type subject_code
Student activity image Required Required students student_code
Staff profile photo within a school Required Required staff staff_code
Org admin profile photo Required Not applicable → _shared staff staff_code
Guardian file Required Usually required, else _shared guardians guardian_code
Org newsletter or general shared doc Required _shared org Not required
School-wide announcement/menu Required Required school Not required

6. Bucket design

Bucket naming

daycare-processed-prod

daycare-raw-staging
daycare-processed-staging

Bucket configuration

  • Versioning enabled
  • Encryption enabled
  • Public access blocked
  • Lifecycle rules enabled

Use one raw and one processed bucket per environment, not a bucket per org or per school. Isolation is enforced with prefixes, DB ownership, IAM, application authorization, and delivery controls.

7. Canonical prefix structure

Raw bucket structure

raw/org/{org_code}/school/{school_code_or_shared}/{subject_type}/{subject_code_or_shared}/{category}/{yyyy}/{mm}/{dd}/{file_code}/original/{sanitized_filename}

Processed bucket structure

processed/org/{org_code}/school/{school_code_or_shared}/{subject_type}/{subject_code_or_shared}/{category}/{yyyy}/{mm}/{dd}/{file_code}/{variant}/{filename}

Subject types

students staff guardians org school

Key rules

  • org/{org_code} is always present.
  • school/{school_code} is always present; if not applicable, use school/_shared.
  • file_code is always present and is the unique anchor for file tracking.
  • The final path segment stores a sanitized filename for readability, not uniqueness.

11. Database schema

The database is the source of truth for ownership, authorization, processing status, and all derived assets. The application should never determine file state by listing S3 keys.

Main files table

    file_code                VARCHAR(64) PRIMARY KEY,

    org_code                 VARCHAR(64) NOT NULL,
    school_code              VARCHAR(64) NULL,

    subject_type             VARCHAR(32) NOT NULL,

    category                 VARCHAR(64) NOT NULL,

    related_entity_type      VARCHAR(64) NULL,
    related_entity_code      VARCHAR(64) NULL,
    module_context           VARCHAR(64) NULL,

13. Key builder rules

All S3 keys must be produced by one shared library or service. This should exist in backend code and be reused by upload APIs, processors, and migration scripts.

Raw key builder

buildRawKey(
  org_code,
  school_code,
  subject_type,
  subject_code,
  category,
  created_date,
  file_code,
  sanitized_filename
)

Processed key builder

buildProcessedKey(
  org_code,
  school_code,
  subject_type,
  subject_code,
  category,
  created_date,
  file_code,
  variant,
  filename
)

Rules

  • If school_code is null, emit school/_shared.
  • If the subject does not require a specific subject code, emit _shared in that slot.
  • Lowercase all path segments except the final display filename if preserving case matters.
  • Do not trust client-supplied paths. The server always constructs the key.
  • Do not store uniqueness in filename. Uniqueness belongs to file_code.

Filename sanitization

Input:  "My Kid's Photo (Final).JPG"
Output: "my-kids-photo-final.jpg"

14. Validation policy

Allowed file families

Family Examples Default size guidance
Images jpeg, png, webp, optionally heic 10 MB for profile photos, 20 MB for activity images
Videos mp4, mov, optionally webm 500 MB default, plan-dependent
Documents pdf, doc, docx, xls, xlsx, csv 25 MB default unless policy says otherwise

Validation stages

  1. Presign API validates declared file metadata.
  2. Processor validates actual object metadata and MIME signature again.
  3. Optional malware scan validates document safety before ready state.
Never trust only the browser-reported content type. Re-check server-side during processing.

15. Processing pipelines

Event path

S3 raw bucket → S3 notification → SQS queue → Lambda processor

Router Lambda behavior

  1. Read bucket and key from SQS payload.
  2. Find file record using original_bucket + original_key.
  3. Skip safely if record is already ready or deleted.
  4. Mark file as processing.
  5. Determine file family: image, video, document, other.
  6. Dispatch to specialized processor path.
  7. Write manifest and final metadata.
  8. Mark ready or failed.

Image pipeline

  • Use Lambda with Sharp or equivalent.
  • Generate avatar sizes for profile photos.
  • Generate gallery-friendly sizes for activity media.
  • Normalize EXIF orientation and strip unnecessary metadata.
  • Store WebP by default, optionally AVIF later.

Video pipeline

  • Router Lambda submits a MediaConvert job.
  • Store job code in DB.
  • MediaConvert writes outputs to processed bucket.
  • Completion handler updates manifest with video variants and poster frame.

Document pipeline

  • Preserve original.
  • Generate preview image or lightweight preview PDF.
  • Optionally extract text or OCR JSON for search and indexing.

Image recommendation

Use Lambda directly. This is ideal for thumbnails, resizing, compression, and WebP delivery variants.

Video recommendation

Use Lambda only as orchestrator. Submit serious transcoding work to MediaConvert.

Document recommendation

Preserve originals. Generate previews and extracted text rather than aggressive compression.

Idempotency requirement

All processors must be safe to rerun. Retries and duplicate events should not corrupt state.

16. Security model

Core controls

  • Block all public access on both buckets.
  • Encrypt at rest using SSE-S3 or SSE-KMS.
  • Use CloudFront to serve processed files.
  • Do not expose raw bucket directly to end users.
  • Authorize every read using org, school, subject, and related entity checks.
  • Issue presigned upload URLs only for one exact key and short expiry.
  • Keep upload audit fields: uploader code, role, timestamps, client source if needed.

Access logic

Requester Can access if Typical scope
Org admin Belongs to org and role policy allows file scope Org-wide, often including school-shared
School admin Belongs to school and policy allows school subject/entity School-level + student/staff/guardian under that school
Staff Belongs to school or org and relationship policy allows view Own media, assigned students, school materials
Guardian Relationship to student is valid Their own files and their child's permitted files
Do not authorize access based only on the S3 key prefix. The database is the policy checkpoint.

Reliability patterns

  • Idempotent workers
  • Deterministic output paths

  • Alerting on repeated failures or MediaConvert backlog

18. Lifecycle and retention

Raw bucket lifecycle

  • Keep recent originals in Standard storage.
  • Transition older originals to Standard-IA or Glacier based on access patterns.
  • Do not auto-delete unless product and compliance policy explicitly allow it.

Processed bucket lifecycle

  • Keep active thumbnails, previews, and common variants hot.
  • Expire temporary intermediates automatically.
  • Delete or regenerate derivatives when parent file is replaced or purged.

Delete policy

  1. Soft-delete in DB first.
  2. Mark object as hidden in product APIs.
  3. Background purge S3 objects if retention rules allow physical deletion.

19. Step-by-step implementation plan

Phase 1 — foundation

  1. Freeze enums: subject types, categories, file families, statuses, variants.
  2. Create raw and processed buckets per environment.
  3. Create the files table and indexes.
  4. Build shared key builder and filename sanitizer utilities.
  5. Define upload policy limits by category and role.

Phase 2 — upload path

  1. Build POST /v1/files/presign-upload.
  2. Create DB row before issuing presigned URL.
  3. Implement frontend direct upload.
  4. Optionally add explicit upload confirmation endpoint.

Phase 3 — processing

  1. Configure S3 raw bucket notifications to SQS.
  2. Build router Lambda.
  3. Implement image processor Lambda.
  4. Implement document preview processor.
  5. Integrate MediaConvert submission and completion handling.

Phase 4 — delivery

  1. Put CloudFront in front of processed bucket.
  2. Implement file metadata and file listing APIs.
  3. Return allowed variant URLs based on authorization.

Phase 5 — hardening

  1. Add malware scan and quarantine flow.
  2. Add dashboards, alarms, and DLQ handling.
  3. Add lifecycle policies and deletion workflows.
  4. Add reprocessing support when image/video rules change.
